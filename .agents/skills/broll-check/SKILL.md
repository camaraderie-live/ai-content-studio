# B-ROLL 素材健檢

開始剪片前，先幫使用者把 `B-ROLL/` 裡的素材檢查一遍：**每段夠不夠短、檔名夠不夠清楚**。這兩件事沒顧好，AI 剪出來的成品品質會明顯變差——所以要先講清楚、讓使用者決定要不要先整理。

## 什麼時候套用

- 盤點素材（Inventory）時、或準備提剪輯策略前。
- 只要專案資料夾裡有 `B-ROLL/` 且裡面有檔案，就主動跑一次這個檢查。
- 使用者新丟了 B-roll 進來時，也重跑一次。

## 兩個檢查項目

### 1. 長度：每段 B-roll 最好不要超過 5 秒

用 `ffprobe` 讀每個影片檔的長度：

```bash
# 用 find -iname：大小寫都吃（iPhone 拍的是大寫 .MOV），資料夾空了也不會報錯
find "<videos_dir>/B-ROLL" -maxdepth 1 -type f \
  \( -iname '*.mp4' -o -iname '*.mov' -o -iname '*.webm' -o -iname '*.mkv' -o -iname '*.avi' -o -iname '*.m4v' \) \
  | while IFS= read -r f; do
  dur=$(ffprobe -v error -show_entries format=duration -of csv=p=0 "$f")
  printf '%6.1fs  %s\n' "$dur" "$(basename "$f")"
done
```

超過 5 秒的，**列出來問使用者要不要自己先剪短到最精華的片段**（不要自己動手剪、也不要自己腦補要留哪段）。跟他說明：B-roll 是疊在主線上的輔助畫面，短、精準的片段疊起來才乾淨；太長的片段 AI 不知道你要用哪一段，容易挑錯或蓋太久，成品會拖。

靜態圖片（`.png` / `.jpg` / `.jpeg` / `.webp` / `.heic` / `.svg` 等）沒有長度問題，跳過長度檢查（命名檢查仍要做）。

### 2. 命名：檔名要清楚說明這段在幹嘛

好的命名讓 AI 直接從檔名就知道這素材是什麼，**不用再去讀影片畫面猜內容**，對得又快又準。

- **好**：`2. 洞察報告截圖.png`、`3. 產品開箱特寫.mp4`
- **不好**：`IMG_4821.MOV`、`螢幕錄製 2026-07-14.mp4`、手機自動產生的一長串英數字

檢查每個檔名：是不是數字開頭（標順序）＋一句話說明內容。不符合的列出來，問使用者要不要改成清楚的名字（可以幫他建議一個，但**實際內容是什麼要他確認**，不要自己看畫面就直接改名）。

## 檢查完怎麼講給使用者聽

用白話總結，明確點出風險，讓使用者自己決定要不要先整理：

> 我看了一下你的 B-ROLL：
> - ⚠️ `X.mp4` 有 12 秒、`Y.mov` 有 8 秒，超過 5 秒了——建議你先自己剪成最精華的 5 秒內片段再給我，疊起來效果會好很多。
> - ⚠️ `IMG_4821.MOV` 這個檔名看不出內容，要不要改成像「3. 產品開箱特寫.mp4」這種清楚的名字？
> - ✅ 其他幾個長度、命名都 OK。
>
> **老實說，如果 B-roll 太長或命名不清楚，我很可能會剪出不理想的成品**（挑錯片段、蓋太久、對錯素材）。先花幾分鐘整理，換來的成品品質差很多。要先整理，還是就這樣直接開始？

**這是提醒，不是強制**——使用者說「就這樣開始」就照做，但風險要先講明白。整理完（或使用者決定不整理）之後，回到正常剪片流程。
